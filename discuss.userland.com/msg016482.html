Well... since you asked. ;->

One of the messages in my inbox is a multipart/mixed with 2 parts. The first part is a multipart/alternative with 2 parts: text/plain, text/html.

The MIME parsing in tcp.getMail doesn't account for this nesting. It only handles MIME parts that are 1 level deep. If that's all it's supposed to do, then I guess it's ok.

The other thing I noticed, not related to MIME, was the presence of the response code from the TCP call. Are you deliberately leaving in the first line with the number of octets returned? If so, I was wondering what functionality this served?
